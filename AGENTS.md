# AGENTS.md — Cyrus Learning Manager

## Repository purpose

Cyrus Learning Manager is a local-first learning manager for Cyrus Knowledge. The static React app can be deployed to GitHub Pages, while a local Express sync service writes learning tasks into SQLite, Obsidian Markdown, and optionally Notion.

Treat this repository as both a software project and a personal learning operating system. Code changes should preserve the learning-loop idea: every feature should help convert study/work inputs into concrete tasks, evidence, notes, and reviewable outputs.

## Tech stack

- Language: TypeScript with strict mode.
- Frontend: React 18, Vite, CSS, Testing Library.
- Backend/local sync: Express, CORS, better-sqlite3, dotenv, Notion SDK.
- Test runner: Vitest with jsdom.
- Deployment: GitHub Pages workflow builds the Vite app from `main`.

## Directory map

- `src/domain/`: shared domain types, learning tracks, knowledge modules, seed tasks.
- `src/api/client.ts`: frontend client for `http://127.0.0.1:8787`.
- `src/components/`: dashboard, task list, course view, progress view, sync center.
- `server/`: local sync service, config loading, SQLite schema, task operations, routes.
- `server/sync/`: Obsidian Markdown writer, Notion database sync, sync event recording.
- `tests/domain/`: domain/knowledge tests.
- `tests/frontend/`: React component tests.
- `tests/server/`: config, auth, database, task, and sync tests.
- `docs/superpowers/`: existing planning/specification material.
- `docs/codex/`: Codex-facing repo map, learning plan, backlog, and review checklists.

## Setup commands

Use npm because this repository already has `package-lock.json`.

```bash
npm ci
npm run dev:web        # Vite app on 127.0.0.1:5173
npm run dev:sync       # local sync service on 127.0.0.1:8787 by default
npm run seed:knowledge # seed SQLite with tracks and learning seed tasks
npm run test           # Vitest test suite
npm run build          # TypeScript no-emit check + Vite build
npm run preview        # preview built Vite app on 127.0.0.1:4173
```

When you modify code, run the narrowest relevant test first, then `npm run test`, then `npm run build` before proposing the final diff.

## Environment and secrets

- Use `.env.local` for local values; never commit real tokens, local vault paths, SQLite data, or generated Obsidian content.
- `.env.example` defines: `NOTION_TOKEN`, `NOTION_PARENT_PAGE_ID`, `NOTION_TASKS_DATABASE_ID`, `OBSIDIAN_VAULT_PATH`, `LOCAL_SYNC_PORT`, `LOCAL_APP_KEY`.
- If the frontend is served from GitHub Pages, write operations must require `LOCAL_APP_KEY` through `x-cyrus-app-key`.
- The sync service should stay bound to `127.0.0.1`; do not expose it on `0.0.0.0` without an explicit security review.
- Tests must use `:memory:` SQLite databases or temporary directories for filesystem sync.

## Coding conventions

- Keep domain model changes centralized in `src/domain/types.ts` first. If you add a `TrackId`, `TaskStatus`, or `TaskPriority`, update frontend filters, `server/tasks.ts` validation, seed data, and tests together.
- Prefer small pure functions for domain and sync formatting logic. Keep Express route handlers thin.
- Use stable, user-facing error messages in the frontend, and detailed assertions in tests.
- Do not introduce new production dependencies unless they clearly remove repeated logic or improve safety. If adding one, explain why it is needed.
- Prefer Zod or equivalent schema validation for request bodies and environment/config validation; the dependency already exists.
- Format TypeScript/TSX into readable multi-line files. Avoid long single-line source files.
- Preserve the local-first design: the app should remain useful without Notion configured, and Notion sync should degrade to `skipped` rather than blocking Obsidian/SQLite.

## Sync-specific rules

- Obsidian writes must be deterministic, safe for repeated syncs, and must not overwrite user-authored notes unless the file is clearly generated by this app.
- Notion sync must validate database schema expectations: `Name`, `Track`, `Status`, `Priority`, `Due`, `Progress`, `Source`, `Obsidian Path`, `Updated At`.
- Sync event recording should capture target, status, message, attempt count, and timestamps.
- When a task is synced to Notion after Obsidian writes, ensure the Obsidian Markdown can be updated with the Notion page id without losing task content.
- Any change to sync behavior needs tests for success, skipped config, failure, and idempotent repeat runs.

## Testing expectations

- Domain changes: add or update tests under `tests/domain/`.
- Frontend changes: add or update Testing Library tests under `tests/frontend/`.
- Server/task/config/sync changes: add or update tests under `tests/server/`.
- Avoid live Notion calls in tests; mock the Notion client or isolate mapping/schema logic.
- Avoid writing to the real Obsidian vault in tests; use a temporary directory.
- After changing env/config behavior, verify `.env.example` and docs stay aligned.

## Review guidelines

Before accepting a diff, check:

- No secrets, personal local paths, tokens, database files, or generated vault content are committed.
- `npm run test` and `npm run build` are expected to pass.
- The local service remains local-only unless explicitly reviewed.
- GitHub Pages origin and local app-key behavior are not weakened.
- Sync changes handle Notion missing config, Obsidian missing path, and partial failures.
- UI changes preserve disconnected-sync behavior and do not hide actionable errors.
- New learning tracks/tasks have clear outputs, source links, and evidence/notes fields.
- Documentation in `docs/codex/` or the root README is updated when workflows change.

Use `docs/codex/code-review.md` as the detailed checklist.

## Learning-loop guidance

When Codex implements or reviews a change, also consider the learning output:

1. What concept did this change exercise? TypeScript strictness, React state, Express routing, SQLite schema, Notion API, Obsidian Markdown, testing, or deployment.
2. What evidence proves progress? A passing test, a built artifact, a synced note, or a clearer task/backlog entry.
3. What should be logged as the next learning task? Keep it small, measurable, and connected to one of the existing tracks.

Use `$cyrus-learning-roadmap` for learning-plan updates, `$cyrus-sync-development` for sync/backend work, `$cyrus-testing-review` for test/review passes, and `$cyrus-repo-onboarding` when the task starts with understanding or planning.
